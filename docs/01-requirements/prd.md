# Product Requirements Document — KPM

> Version: v0.1  
> Status: Draft for review  
> Current target release: V1 可试点版

## 1. Executive summary

KPM（Kozen Project Management）是 Kozen 内部使用的产品项目管理系统，用于管理 POS 产品型号从想法提出到客户维护的全生命周期。系统的首要目标，是把当前分散在聊天工具和不同任务平台中的信息、阶段状态、责任人和组织关系统一起来，让公司成员能够在权限范围内看到某个产品项目的阶段状态、参与人员和推进情况。

V1 将被定义为“可试点版”，不是单纯的后台管理系统。除用户、角色、部门、权限、项目等基础能力外，V1 还需要支持流程模板、项目成员配置、项目阶段展示以及阶段负责人配置，使部分团队可以开始真实试用。

## 2. Background and problem statement

Kozen 的核心业务是生产和销售 POS 机，客户分布于全球多个国家。一个产品型号从构想到上市，需要跨越多个部门和阶段，当前参考流程包括：提出想法、讨论可行性、核算成本、硬件设计、软件适配、测试生产、试用、客户推广、客户调试、客户下单、发货给客户、客户维护。

当前协作高度依赖聊天工具，不同团队又分别使用 Jira、Teambition 或口头沟通进行任务管理，导致：

- 项目信息分散，缺乏统一可信源
- 项目所处阶段和责任人不透明
- 新入职成员难以快速找到正确协作对象
- 旧信息继续传播，更新无法同步到所有相关团队
- 跨部门协同成本高，项目管理依赖个人经验

## 3. Product goals and success metrics

### Goals

1. 建立一个公司级统一项目视图
2. 让项目流程、各阶段状态和责任人透明化
3. 让组织、角色和项目成员关系可配置、可维护
4. 为未来的信息共享、审批、延期、通知和第三方集成提供稳定基础

### Initial success metrics for V1

- 试点团队可以在系统中创建和维护项目
- 每个项目都能查看完整流程以及各阶段状态
- 每个阶段都能明确显示负责人或负责部门
- 用户、部门、角色、项目内角色可以被管理员正确维护
- 新成员无需依赖口头询问即可找到项目相关责任人

## 4. Users and personas

| Persona | Main needs |
| --- | --- |
| 管理员 | 管理用户、部门、角色、权限、流程模板 |
| 项目创建人 / 项目负责人 | 创建项目、选择成员、配置项目流程与责任人 |
| 硬件项目经理 / 成员 | 查看项目阶段、确认硬件相关责任和状态 |
| 软件项目经理 / 成员 | 查看项目阶段、确认软件相关责任和状态 |
| 技术支持团队 | 了解面向客户阶段所需的上游信息与责任人 |
| 销售团队成员 | 跟踪产品上市进度、识别当前负责人 |
| 运营 / 测试团队成员 | 查看自身参与项目的阶段安排和协作对象 |

## 5. Scope

### In scope for V1

1. **用户管理**
   - 新增、编辑、停用、查询用户
   - 用户可属于多个部门
   - 用户可直接持有少量例外权限

2. **部门管理**
   - 新增、编辑、删除、查询部门
   - 支持一个用户关联多个部门
   - 部门成员数由用户归属自动统计

3. **角色管理**
   - 新增、编辑、删除、查询角色
   - 支持全局角色
   - 支持项目内角色
   - 支持在角色中统一配置权限

4. **权限管理**
   - 权限由系统根据真实菜单和真实按钮自动生成
   - 权限分为菜单权限与按钮权限
   - 权限管理页仅支持查看和按类型筛选，不支持手工新增、编辑、删除
   - 支持菜单可见权限，左侧菜单需配置后才可见
   - 权限授权只支持两种路径：角色授权、用户直授

5. **项目管理**
   - 创建、编辑、查询、归档项目
   - 已归档项目需要支持恢复
   - 维护对外名称、内部名称、Model 名称等基础字段
   - 每个项目都需要维护“销售状态”：可销售 / 不可销售
   - 不可销售需要记录原因：仍在设计或测试、产品过老不再推广、被遗弃的老项目
   - 项目列表支持按销售状态筛选
   - 项目列表支持按归档状态筛选
   - 项目负责人应引用真实用户
   - 项目负责人应自动进入项目成员列表
   - 创建项目时从真实用户中选择项目成员
   - 为成员分配项目内角色

6. **流程模板管理**
   - 创建、编辑、复制、启用、停用流程模板
   - 支持删除草稿或已停用模板；启用模板需先停用后再删除
   - 为模板配置多个阶段
   - 阶段顺序可调整

7. **项目阶段管理**
   - 项目基于流程模板生成阶段列表
   - 展示完整阶段路径以及每个阶段的独立状态
   - 每个阶段可绑定一个或多个真实负责人 / 负责部门
   - 阶段之间可并行进行，不以单一“当前阶段”作为唯一推进依据
   - 需要支持受权限控制的阶段状态维护

8. **客户管理**
   - 系统需要支持客户基础资料新增、编辑、删除、查询
   - 客户维护属于受控权限，可通过角色授权或用户直授配置
   - 客户基础资料与项目下客户状态应区分建模
   - 客户基础资料需要支持配置多个负责销售和多个负责技术支持

9. **项目客户管理**
   - 项目详情页需要展示与当前产品项目关联的客户列表
   - 同一客户可被关联到多个项目，并在不同项目下拥有不同状态
   - 每个项目客户关系需要维护客户当前状态，例如推广阶段、稳定出货阶段
   - 项目客户状态不需要流程模板，只需要由有权限人员维护可选枚举值
   - 项目成员可按权限将已有客户关联到项目并维护客户在该项目下的状态
   - 在项目客户列表中，需要能进入该客户的需求列表

10. **客户需求管理**
   - 每个项目下的每个客户都可以维护自己的需求列表
   - 需求记录格式参考敏捷开发中的需求记录方式
   - 单条需求至少包含：标题、用户故事、业务价值、验收标准、优先级、状态、提出人、提出日期
   - 需求列表需要支持新增需求
   - 需求记录需要支持作废；删除仅用于误录或不应保留的记录
   - 创建需求时，应支持同时创建关联任务
   - 关联任务默认流转给需求创建者
   - 需求上需要记录关联任务 ID，并支持跳转到任务详情
   - 当关联任务完成时，需求自动标记为“已实现”
   - 当关联任务被拒绝时，需求自动标记为“已拒绝”
   - 项目详情页需要提供“需求纵览”，用于查看多个客户的共性需求，方便产品提取和抽象


11. **阶段详情与协作**
   - 每个项目阶段都需要可进入独立详情页
   - 阶段详情本身作为权限边界：能进入阶段详情的人，默认可以查看和下载该阶段资料
   - 阶段详情页包含阶段资料库，可上传和下载任意格式文件，包括图片、视频等
   - 阶段详情页包含类似论坛留言的信息流
   - 阶段负责成员可发布留言或记录信息
   - 留言可包含文字、图片、视频或其他格式附件
   - 留言与记录需按时间排序展示

12. **项目资料区**
   - 阶段资料默认不再单独配置文件级权限
   - 需要跨阶段复用的资料，由有权限的阶段负责人主动“发布到项目资料区”
   - 项目资料区用于沉淀项目内可复用的可信资料，并展示来源阶段与发布时间

13. **国际化支持**
   - 系统需要支持中文和英文
   - 首期仅支持中英文，不支持其他语种

14. **任务管理**
   - 系统需要提供独立任务管理菜单
   - 任务可在任务管理页直接创建，也可从阶段详情页创建
   - 任务可关联项目和来源阶段
   - 新建任务时需要自动记录任务创建者
   - 新建任务时需要选择任务分类，首期分类包括：需求、Bug、技术支持、其他
   - 当前定位是记录和管理项目成员的日常任务，不以替代 Jira 为目标
   - 任务管理参考 Jira 的基础字段设计，包括任务分类、任务状态、优先级、执行者、参与者、预期完成时间、筛选检索等能力
   - 任务执行者和参与者都支持多人，并支持按姓名输入搜索
   - 阶段详情需要支持“卡点求助”，本质上生成一条任务并流转给阶段负责人，同时将项目负责人加入参与者
   - 暂不单独建设延期申请流程；阶段延期先通过任务截止日期和任务跟进来体现
   - 系统需要提供任务详情页，用于查看并维护任务描述、优先级、执行者、参与者、预期完成时间、关联对象和状态流转
   - 任务详情页需要支持上传照片、视频及其他格式附件
   - 任务详情页需要支持评论 / 留言，留言可附带附件，用于执行者和参与者围绕任务协作
   - 任务状态需要支持配置，并支持定制允许的状态流转关系
   - 暂不纳入看板、Sprint、Epic、Story 等重型研发管理能力

15. **订单管理**
   - 系统需要提供订单管理菜单
   - 订单至少包含：下单日期、下单客户、项目信息、数量、具体规格、期望发货日期、计划发货日期、软件版本号、币种、单价、订单金额、创建人
   - 订单金额默认由 `数量 × 单价` 计算
   - 订单需要支持新增、编辑、删除、查询
   - 当订单被修改时，系统需要记录修改人、修改时间、修改内容和修改原因
   - 订单新增、编辑、删除、查询需要纳入权限配置

16. **统计看板**
   - 系统需要提供统计看板菜单，并拆分为不同二级菜单
   - 订单统计需要支持按年、季度、月份，统计一个或多个项目的销售额、订单数量和产品数量
   - 资源分配情况需要按地区展示客户、负责销售、负责技术支持和已售项目，并支持通过拖拽地图查看全球资源分布
   - 技术支持情况需要支持按客户查询当前负责技术支持、进行中任务和卡点任务
   - 技术支持情况建议以树状图展示：客户 → 技术支持 → 当前任务统计，并支持悬浮查看摘要、点击查看任务详情

17. **登录与退出**
   - 系统需要提供登录页和登录逻辑
   - 登录页需要采用四个小人跟随鼠标、输入密码时捂眼睛的互动设计
   - 登录后，点击右上角用户名需要展示退出登录入口

18. **Web 端支持**
   - 首期只支持 Web

### Out of scope for V1

- 审批流
- 单独的阶段延期申请流程
- Jira 集成
- 钉钉通知
- 复杂信息共享策略
- 移动端 App
- 高级报表分析（首期先实现已明确的统计看板）

## 6. Core user journeys

### Journey A — 管理员初始化组织结构

1. 管理员创建部门
2. 管理员创建用户
3. 管理员将用户关联到一个或多个部门
4. 管理员创建角色并配置权限
5. 管理员维护流程模板

### Journey B — 创建一个新产品项目

1. 项目创建人进入项目管理
2. 填写项目基本信息
3. 选择适用流程模板
4. 选择项目成员
5. 为项目成员分配项目内角色
6. 生成项目阶段列表
7. 为阶段配置负责人或负责部门
8. 项目创建完成

### Journey C — 查看项目当前状态

1. 用户进入项目列表
2. 搜索某个产品型号
3. 打开项目详情
4. 查看完整流程、各阶段状态、阶段责任人和项目成员
5. 按权限查看对应信息

## 7. Functional requirements

### FR-01 用户管理

- 系统应支持用户新增、编辑、停用、查询
- 系统应支持一个用户关联多个部门
- 系统应支持为具体用户配置用户直授权限

### FR-02 部门管理

- 系统应支持部门新增、编辑、删除、查询
- 部门成员数应根据用户所属部门自动统计

### FR-03 角色管理

- 系统应支持角色新增、编辑、删除、查询
- 系统应支持全局角色与项目内角色
- 系统应支持在角色中配置菜单权限与按钮权限

### FR-04 权限管理

- 系统应根据系统中真实存在的菜单和按钮自动生成权限清单
- 权限应分为菜单权限与按钮权限
- 权限管理页应仅支持查看和按类型筛选，不支持手工新增、编辑、删除
- 系统应支持菜单可见权限配置
- 未被授予菜单权限的用户，不应在左侧导航中看到对应菜单
- 系统应支持通过角色授权或用户直授两种方式赋权
- 部门仅用于组织归属，不直接承载权限
- 系统应根据权限控制用户可访问页面和操作

### FR-05 项目管理

- 系统应支持项目新增、编辑、查询、归档
- 系统应支持已归档项目恢复
- 项目至少包含：对外名称、内部名称、Model 名称、描述、状态、销售状态、创建人
- 销售状态应支持“可销售 / 不可销售”
- 不可销售时应支持记录原因：仍在设计或测试、产品过老不再推广、被遗弃的老项目
- 项目列表应支持按销售状态筛选
- 项目列表应支持按归档状态筛选
- 项目负责人应引用真实用户
- 项目负责人应自动进入项目成员列表
- 创建项目时可从真实用户中选择项目成员
- 可为项目成员分配项目内角色

### FR-06 流程模板管理

- 系统应支持维护流程模板
- 系统应支持流程模板新增、查询、编辑、复制、启用、停用
- 系统应支持删除草稿或已停用模板；启用模板需先停用后再删除
- 流程模板应包含多个可排序阶段
- 阶段可新增、编辑、删除、排序

### FR-07 项目阶段展示与维护

- 系统应根据流程模板生成项目阶段
- 系统应展示完整阶段路径以及每个阶段的独立状态
- 阶段之间可以并行进行，不要求严格串行
- 每个阶段应可绑定一个或多个真实负责人 / 负责部门
- 系统应支持有权限的阶段负责人维护其负责阶段的状态

### FR-08 客户管理

- 系统应支持客户基础资料新增、编辑、删除、查询
- 系统应支持通过角色授权或用户直授控制客户录入与维护权限
- 客户基础资料至少应支持客户名称、国家 / 区域、负责销售、负责技术支持与状态等基础信息
- 负责销售和负责技术支持都应支持多选

### FR-09 项目客户管理

- 系统应在项目详情页展示与当前项目关联的客户列表
- 系统应支持将已有客户关联到某个项目
- 同一客户在不同项目下可以拥有不同状态
- 系统应支持维护客户在当前项目下的状态，例如推广阶段、稳定出货阶段
- 系统应支持由有权限用户维护项目客户状态枚举值
- 系统应支持从项目客户列表进入该客户在当前项目下的需求列表

### FR-10 客户需求管理

- 系统应支持在项目客户关系下维护需求列表
- 需求记录应支持至少：标题、用户故事、业务价值、验收标准、优先级、状态、提出人、提出日期
- 系统应支持新增需求
- 系统应支持将需求标记为作废
- 系统应支持删除误录或不应保留的需求
- 系统应支持在创建需求时同步创建关联任务
- 关联任务应默认分配给需求创建者
- 需求记录应保存关联任务 ID，并支持跳转到任务详情
- 当关联任务状态为“已完成”时，需求状态应自动更新为“已实现”
- 当关联任务状态为“已拒绝”时，需求状态应自动更新为“已拒绝”
- 项目详情页应提供需求纵览，用于查看和归纳多个客户的共性需求

### FR-11 阶段详情与资料协作

- 系统应支持从项目详情进入具体阶段详情页
- 阶段详情应作为资料访问权限边界
- 能进入阶段详情的用户，应默认可查看和下载该阶段资料
- 阶段详情页应包含阶段资料库
- 阶段资料库应支持上传和下载任意格式文件
- 阶段详情页应包含按时间排序的信息流
- 阶段负责成员应可发布文字留言，并可附带图片、视频或其他格式文件

### FR-12 项目资料区

- 系统应支持将资料发布到项目资料区
- 项目资料区应展示资料名称、来源阶段、发布目标与发布时间

### FR-13 国际化

- 系统应支持中文和英文两种语言
- 系统界面文案应支持语言切换
- 首期不支持其他语种

### FR-14 任务管理

- 系统应提供任务管理页面
- 系统应提供任务详情页面
- 用户应可在任务管理页直接新建任务
- 用户应可从阶段详情页发起任务
- 用户在创建任务时应选择任务分类
- 首期任务分类应至少包含：需求、Bug、技术支持、其他
- 新建任务时系统应自动记录创建者
- 阶段详情页应支持发起卡点求助任务
- 卡点求助任务应自动流转给阶段负责人，并自动将项目负责人加入参与者
- 任务应支持至少：标题、任务分类、所属项目、来源阶段、状态、优先级、创建者、执行者、参与者、预期完成时间
- 任务执行者和参与者都应支持多人，并支持按姓名输入搜索
- 任务详情应支持查看关联需求，并在任务状态变化时向关联需求回写状态
- 任务详情应支持维护任务描述、优先级、执行者、参与者和预期完成时间
- 任务详情应支持上传照片、视频及其他格式附件
- 任务详情应支持评论 / 留言，留言可附带附件
- 任务列表应支持按关键字、项目、状态、优先级、执行者进行筛选
- 系统应支持配置任务状态，并支持配置允许的状态流转关系
- 阶段延期暂不建设独立流程，先通过任务截止日期和任务跟进体现
- 当前任务模块定位为日常任务记录与管理，不要求看板、Sprint、Epic 或 Story 能力

### FR-15 订单管理

- 系统应提供订单管理页面
- 订单应支持新增、编辑、删除、查询
- 订单至少应包含：下单日期、下单客户、项目信息、数量、具体规格、期望发货日期、计划发货日期、软件版本号、币种、单价、订单金额、创建人
- 订单金额应默认按 `数量 × 单价` 计算
- 修改订单时，系统应保存修改人、修改时间、修改内容和修改原因
- 订单新增、编辑、删除、查询应支持权限配置

### FR-16 统计看板

- 系统应提供统计看板页面，并支持订单统计、资源分配情况、技术支持情况三个二级菜单
- 订单统计应支持按年、季度、月份与一个或多个项目筛选
- 订单统计应展示销售额、订单数量、产品数量
- 资源分配情况应按地区展示客户、负责销售、负责技术支持和已售项目
- 资源分配情况应支持拖拽地图查看全球区域
- 技术支持情况应支持按客户查询技术支持负责人、进行中任务和卡点任务
- 技术支持情况应支持以树状图展示客户、技术支持和任务统计，并支持查看任务详情

### FR-17 登录与退出

- 系统应提供登录页与登录逻辑
- 登录页应具备小人跟随鼠标、输入密码时捂眼睛的互动效果
- 登录后应支持从右上角用户名入口退出登录

### FR-18 基础查询

- 用户可按项目名称、内部名称、Model 名称进行检索
- 用户可查看与自己相关的项目

## 8. Non-functional requirements

### Performance and capacity

- 系统需支持至少 100 名用户同时在线使用
- 常用页面在正常网络下应具备良好响应体验
- 后续技术设计需支持高并发扩展能力

### Availability

- 设计上应考虑高可用部署能力
- 关键服务应避免单点故障

### Security

- 系统应具备身份认证与权限控制
- 权限模型应支持未来扩展到更细粒度的数据访问控制

### Usability

- 重要操作路径应清晰，尤其是项目创建和项目状态查看
- 用户应能在较少学习成本下找到“某个阶段由谁负责”

### Compatibility

- 首期支持主流桌面浏览器上的 Web 使用

### Internationalization

- 系统从 V1 起支持中文与英文
- 界面文案、提示语和状态标签需要具备国际化兼容能力

## 9. Permissions and role model

### Confirmed model

KPM 需要同时支持：

1. **全局角色**：决定系统级能力，例如管理员
2. **项目内角色**：决定用户在某个项目中的职责，例如某项目中的软件项目经理
3. **多部门归属**：一个用户可以属于多个部门
4. **两层授权模型**：权限通过角色赋予，或直接赋予到具体用户；部门不直接授权

### V1 visibility rule

- 项目基础信息与各阶段状态默认对具备系统访问权限的公司成员可见
- 更敏感的阶段资料、成本资料和技术资料，在 V2 之后通过更细粒度的数据权限控制

### Design implication

V1 可以先实现基础的权限配置能力，但底层数据模型必须支持角色授权与用户直授，并为后续项目级、阶段级、信息级权限扩展预留空间，避免未来重构。

## 12. Risks, assumptions, and dependencies

### Risks

- 如果 V1 只建设后台而不承载真实业务流，试点价值会不足
- 权限模型如果设计过浅，后续信息共享和审批会遭遇重构成本
- 流程配置如果设计过死，会违背业务初衷

### Assumptions

- 一个项目当前对应一个产品型号
- 首期只支持 Web
- 首期可先不实现复杂审批和通知
- 各团队愿意在试点范围内维护项目数据

### Dependencies

- 后续 Jira、钉钉集成依赖相应系统接口和授权条件
