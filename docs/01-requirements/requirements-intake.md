# Requirements Intake

## 1. Product purpose

KPM（Kozen Project Management）用于管理 Kozen 的 POS 产品从想法提出到售后维护的全生命周期，并把分散在聊天工具中的项目信息、阶段状态、负责人和共享资料统一沉淀到一个可信系统中。

### Current problems

- 目前各团队主要通过聊天工具协作，信息分散且难以追溯
- 新入职同事不知道不同问题应该找谁
- 项目信息会发生更新，但其他相关团队不一定及时知道
- 项目推进缺乏一个公司级统一视图
- 各团队使用 Jira、Teambition、口头沟通等不同方式管理任务，协同效率低

### Desired outcomes

- 全公司成员在权限范围内查看某个产品当前所处阶段及详细信息
- 明确每个阶段的负责人或负责部门
- 建立项目可信信息源，减少重复询问和版本混乱
- 支持未来按业务变化配置流程步骤
- 为后续审批、延期、卡点求助、通知、第三方集成打下基础

## 2. Users, departments, and roles

### Known roles

- 管理员
- 硬件项目经理
- 硬件项目成员
- 软件项目经理
- 软件项目成员
- 技术支持团队
- 销售团队成员
- 运营团队成员
- 测试团队成员
- 其他后续可扩展角色

### Confirmed organizational rules

- 角色需要支持按需新增
- 每个角色需要可配置权限
- 左侧菜单需要按权限显示，未授权菜单不可见
- 角色需要支持“跟项目绑定”
- 同一个人在不同项目中可以承担不同角色
- 同一个人可能同时属于多个部门

## 3. Core business object

### Project

一个项目对应一个产品型号。

每个项目还需要维护销售状态：

- 可销售：设计和测试已完成，可推广并允许客户下单
- 不可销售：原因包括仍处于设计 / 测试阶段、产品过老不再推广、被遗弃的老项目

示例：

- 对外名称：P8 dual
- 内部名称：R2351
- Model 名称：K1352

项目立项后进入产品生命周期流程，并推进研发、试产、推广、交付和维护。

## 4. Product lifecycle workflow

### Current reference workflow

1. 提出想法
2. 讨论可行性
3. 核算成本
4. 硬件设计
5. 软件适配
6. 测试生产
7. 试用
8. 客户推广
9. 客户调试
10. 客户下单
11. 发货给客户
12. 客户维护

### Workflow requirements

- 流程不是固定的，需要支持配置
- 后续可新增、删除或调整步骤
- 阶段之间不要求严格串行，多个阶段可能并行进行
- 每个步骤可配置负责人或负责部门
- 不同步骤的负责方可能不同
- 某些步骤的信息需要共享给后续步骤或指定团队
- 阶段资料默认继承阶段详情访问权限，不再额外做文件级权限
- 跨阶段复用通过显式发布动作完成，即发布到项目资料区

## 5. Information sharing and permissions

### Examples

- 硬件设计、软件适配阶段的技术方案、硬件信息，可能需要共享给客户推广之后的所有阶段
- 核算成本阶段的成本信息，不一定对所有人开放，可能只共享给领导团队或销售团队

### Requirements

- 信息需要按权限控制可见性
- 后续需要支持项目、阶段、信息级别的权限策略
- 需要建立“当前可信版本”的信息机制，避免聊天记录中的旧信息继续传播
- 权限除了可配置给角色外，也需要支持配置给具体部门或具体人员
- 例如：客户录入、客户维护可以授权给某个部门或某个具体用户

## 6. Task and collaboration needs

### Current state

- 部分团队使用 Jira
- 部分团队使用 Teambition
- 部分团队依赖口头沟通

### Future expectations

- 项目创建时由创建人选择项目成员
- 系统需要支持客户基础资料录入与维护
- 系统需要提供任务管理模块
- 任务可在任务管理页直接创建，也可从阶段详情中创建
- 新建任务时需要选择任务分类，首期分类包括：需求、Bug、技术支持、其他
- 任务管理方式可参考 Jira，包括任务分类、状态、优先级、负责人、截止日期和过滤查询
- 系统需要提供任务详情页，承载任务描述、关联对象、参与者和状态变化
- 当前任务模块仅用于记录和管理项目成员日常任务，不以替代 Jira 为目标
- 如果可行，未来集成 Jira：
  - 在 KPM 中创建 Jira 任务
  - 通过钉钉通知相应人员

## 7. Approval, scheduling, and issue handling

### Approval example

车间生产前的软件版本确认流程：

1. 车间发起
2. 软件项目经理确认
3. 售前技术支持工程师确认

### Scheduling requirements

- 每个项目的每个阶段都应有独立状态，且由对应阶段负责人维护
- 每个阶段需要有独立详情页
- 阶段详情页需要包含资料库，支持上传和下载任意格式文件
- 阶段详情页需要包含按时间排序的留言 / 记录功能，支持文字、图片、视频和其他附件
- 每个项目的每个阶段都应有预计完成时间
- 当前阶段先不单独建设延期流程
- 若需要体现延期，先通过任务截止日期和任务跟进来承接

### Blocker handling

- 任意项目成员都可以发起卡点求助
- 卡点求助从阶段详情发起，本质上生成一条任务
- 任务自动流转给阶段负责人
- 项目负责人自动加入该任务的参与者

## 8. Initial release expectation

### Confirmed V1 direction: pilot-ready release

V1 需要是“可试点使用版”，不仅包含后台基础能力，也需要让部分团队已经能够真实使用系统查看项目推进情况。

### Proposed V1 scope

- 用户管理 CRUD
- 角色管理 CRUD
- 部门管理 CRUD
- 权限管理 CRUD
- 项目管理 CRUD
- 流程模板管理
- 项目成员配置
- 项目阶段展示
- 阶段负责人 / 负责部门配置
- 项目客户列表与客户状态维护
- 项目销售状态与过滤
- 客户需求列表
- 需求新增、作废、删除
- 创建需求时可同步创建关联任务
- 需求可跳转到关联任务详情
- 任务完成 / 被拒绝时自动回写需求状态
- 项目需求纵览
- 客户基础资料管理
- 客户等级与客户等级枚举配置
- 客户活跃度统计看板
- 项目客户状态枚举配置
- 项目编辑弹窗
- 项目归档 / 恢复 / 删除二次确认
- 客户详情关联项目列表
- 下单时自动建立客户-项目关联，并按订单类型初始化客户项目状态
- 阶段详情、阶段资料库、阶段留言
- 菜单权限
- 任务管理
- 任务详情页
- 项目资料区
- 阶段资料向后续阶段共享
- 中英文双语兼容
- Web 端优先

### Explicitly deferred beyond V1

- 复杂审批流
- 单独的延期申请机制
- Jira 集成
- 钉钉通知
- 更细粒度的信息共享策略

## 9. Known future scope

- 阶段信息共享
- 审批流
- 单独的延期机制
- Jira 集成
- 钉钉通知
- 更完整的消息中心与待办中心

## 10. Open questions

1. 权限在首期是否仅采用角色级别，还是需要预留项目 / 阶段 / 字段级权限？
2. 一个项目是否可能存在多个产品变体，或多个型号之间存在继承关系？
3. 项目生命周期流程是全公司统一模板，还是不同产品线可使用不同模板？
4. V1 是否需要基础的项目资料区，用于沉淀关键文档或版本信息？
5. 订单统计进行币种换算时，后续应采用哪一种正式汇率口径？

## 11. Internationalization

- 系统需要支持中文和英文
- 首期仅支持中英文，不考虑其他语种
- 页面、菜单、按钮、提示文案应从一开始按国际化方式设计，避免后续大量硬编码改造
