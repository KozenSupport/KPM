# KPM 前端按钮与接口联通盘点

> 更新时间：2026-06-01。目标是把原型运行态中的按钮按“真实落库 / 本地草稿 / 仅页面交互 / 待补齐”分类，方便后续逐项收口。

## 1. 已接真实 API，并已接入二次确认

这些操作点击后会先做必填校验，再弹出二次确认；用户确认后才会调用后端接口。

| 页面/模块 | 操作 | 说明 |
| --- | --- | --- |
| 订单管理 | 新增订单、保存订单修改、删除订单 | 删除、修改、提交均二次确认 |
| 客户详情 | 为客户下单 | 使用订单接口落库 |
| 客户管理 | 新增/编辑客户、删除客户 | 删除受使用状态限制 |
| 客户详情 | 新增联系人、删除联系人、提交跟进记录、上传客户资料 | 联系人删除已补后端接口 |
| 任务管理 | 新建任务、保存任务详情、提交评论、上传附件、修改任务状态 | 状态修改二次确认 |
| 阶段详情 | 新建阶段任务、卡点求助、提交阶段留言、上传阶段资料、修改阶段状态 | 上传和状态变更均二次确认 |
| 项目详情 | 修改项目成员、关联客户、修改客户项目状态 | 客户项目状态变更二次确认 |
| 项目编辑 | 保存项目修改 | 项目负责人、成员、阶段负责人等保存前二次确认 |
| 新建项目 | 第四步最终创建项目 | 最终提交前二次确认 |
| 客户需求 | 新增需求、作废需求、删除需求 | 作废/删除有二次确认 |
| 流程模板 | 新增/编辑模板、复制模板、删除模板、启停模板 | 模板草稿阶段编辑仍为本地草稿，保存时落库 |
| 资源管理 | 用户、部门、角色、枚举、客户资源的新增/编辑/删除/启停 | 真实 API，保存/删除/启停均二次确认 |
| 任务状态配置 | 新增/删除状态流转 | 已补资源服务接口 |

## 2. 已有二次确认，并已完成接口成功后刷新

| 操作 | 当前情况 | 说明 |
| --- | --- | --- |
| 项目归档/恢复 | 已有确认弹窗，调用项目接口 | 已改为接口成功后统一刷新后端数据 |
| 项目删除 | 已有确认弹窗，调用项目接口 | 已改为接口成功后再刷新；删除失败时不会本地移除 |
| 阶段资料发布到项目资料区 | 已有确认弹窗，调用发布接口 | 已改为接口成功后刷新项目资料区 |

## 3. 本地草稿操作，不需要立即落库

这些操作只是调整表单里的临时内容，真正落库发生在“保存/提交”按钮，因此暂不需要单独二次确认。

| 操作 | 原因 |
| --- | --- |
| 项目成员弹窗里添加/移除成员 | 只是修改成员表单草稿，保存成员时才落库 |
| 新建/编辑流程模板里新增、删除、上移、下移阶段 | 只是修改模板草稿，保存模板时才落库 |
| 用户/角色权限选择器中添加/移除权限 | 只是修改授权草稿，保存用户/角色时才落库 |

## 4. 纯页面交互，不需要接口

打开/关闭弹窗、查看详情、筛选、切换 tab、返回、地图全屏、查看修改记录、选择模板、查看关联对象等仍然属于页面交互，不应调用后端写接口。

## 5. 下一步建议补齐

1. 做一轮页面级 E2E smoke test：订单、客户、任务、项目、资源、流程模板各选一条完整路径。
2. 把按钮权限真正接入渲染逻辑：无权限时按钮隐藏或禁用，并在接口层继续保留权限校验。
3. 继续减少本地状态手动拼接，所有写操作保持“接口成功后刷新后端数据”的模式。
