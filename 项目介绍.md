## 4. 是否引入 MQ？如何使用？

当前 V1 采用**数据库 Outbox 事件表 + 通知微服务轮询消费**，暂未引入独立 RocketMQ/Kafka/RabbitMQ。

- 业务服务在创建任务、项目、订单等动作后写入 `kpm_notification_events`，相当于轻量消息队列。
- `kpm-notification-service` 定时消费 `PENDING` 事件，生成 `kpm_internal_messages`，并可按 Nacos 中的邮件配置发送邮件。
- 前端登录后会按 Nacos 配置的刷新间隔轮询内部消息，默认 120 秒。
- 这种方式免费、部署简单，适合 V1 试点；后续如果通知量和异步任务量变大，可以将 Outbox 消费器替换为 RocketMQ/Kafka，同时保留事件表作为可靠补偿。

### 5.2 当前是否有签名和鉴权验证？

当前 V1 试点版已实现网关级 Bearer token 校验：除 `/api/iam/login`、健康检查和 Swagger 相关路径外，浏览器 API 请求必须携带 `Authorization: Bearer <token>`。

当前已有的是：

- 登录接口 `/api/iam/login` 返回 HMAC 签名的 KPM Bearer token；前端保存到 `localStorage` 并在后续 API 请求中自动带上。
- 网关校验 token 是否有效、是否过期，并将账号和用户 ID 写入请求头。
- 网关对新增、修改、删除等写操作做真实 RBAC 校验，例如任务保存需要 `button:tasks:save`；没有权限会由后端返回 `403 FORBIDDEN`。
- 业务写接口已在后端做必要字段、长度、邮箱、日期、数字、枚举、用户选择有效性等基础校验。

正式上线前建议继续补齐：

1. 接口防重放、CSRF、审计日志和数据范围校验；
2. 更细粒度的数据权限，例如只能修改自己负责项目/客户下的数据；
3. token 刷新、强制下线和密码策略。

## 7. 后端代码如何设计和分层

后端采用 Java 21 + Spring Boot 4 微服务结构。当前每个业务模块是一个独立服务：

```text
apps/backend/
  kpm-gateway/              API 网关
  kpm-iam-service/          登录、当前用户、权限聚合
  kpm-resource-service/     用户、部门、角色、权限、枚举、状态流转
  kpm-project-service/      项目、阶段、流程模板、需求、项目资料
  kpm-customer-service/     客户、联系人、客户资料、跟进记录
  kpm-task-service/         任务、执行者、参与者、附件、评论
  kpm-order-service/        订单、订单修改记录
  kpm-file-service/         OSS 文件上传和下载链接
  kpm-analytics-service/    统计看板、资源地图、客户活跃度
  kpm-common/               公共响应、JDBC 行转换、ID、JSON、SQL、校验、OpenAPI 配置
```

每个业务服务按以下层次组织：

| 层 | 责任 |
| --- | --- |
| `controller` | 接收 HTTP 请求，声明 REST API 和 Swagger 文档，不写业务 SQL |
| `service` | 定义业务接口，表达业务能力 |
| `service.impl` | 业务编排、事务控制、数据校验、跨表联动 |
| `mapper` | JDBC SQL 访问，负责具体查询和写库 |
| `common` | 复用工具、API 响应、校验、OpenAPI 配置 |

当前为了快速试点，接口 payload 仍大量使用 `Map<String,Object>`，后续进入更正式开发阶段时，建议逐步替换为 DTO + Bean Validation 注解，以增强类型安全和 Swagger 表达能力。
